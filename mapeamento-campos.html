Documentação Interface Mapeamento de campos Estrutura JSON – Ticket HugMe
Este documento detalha o mapeamento entre os campos retornados pela API Hugme (ReclameAQUI) e os objetos e campos do Salesforce. A lógica de transformação está nas classes IntegrationProductHugMeDTO (estrutura de dados da API) e IntegrationProductHugMeDomain (criação/atualização de registros no Salesforce).

1. Ticket (TicketBO) → Case

Cada ticket da API é tratado como um Case. O Case é criado em IntegrationProductHugMeDomain.createCase(TicketBO) e atualizado em getCaseByHugMeId (status HugMe e RA).

Campo Hugme (DTO / API) Objeto / Campo Salesforce Observação
TicketBO.idCase.HugMe_Ticket_Id__cIdentificador do ticket no Hugme.
TicketBO.hugme_status.nameCase.Status_HugMe__cStatus no Hugme (ex.: "Novo"). Atualizado se o Case já existir.
TicketBO.ra_status.nameCase.Status_RA__cStatus no ReclameAQUI (ex.: "Não respondido"). Atualizado se o Case já existir.
TicketBO.creation_dateCase.Data_de_Criacao_Hugme__cData de criação do ticket no Hugme.
TicketBO.last_modification_dateCase.Ultima_Modificacao_Hugme__cÚltima modificação do ticket no Hugme.
TicketBO.interactions[0].messageCase.Manifestacao_Reclame_Aqui__cMensagem da primeira interação (manifestação).
TicketBO.interactions[0].message ou texto padrãoCase.DescriptionDescrição do caso; fallback: "Ticket criado a partir da HugMe".
—Case.SubjectFixado: "Ticket HugMe #" + ticket.id.
—Case.StatusFixado: "New".
—Case.OriginFixado: "Reclame Aqui".
—Case.PriorityFixado: "Medium".
TicketBO.customer (objeto criado)Case.Cliente__cLookup para ClientsHugmeIntegration__c (ver seção 2).
—Case.RecordTypeIdRecord Type com DeveloperName RA_da_Ra.

Campos do TicketBO não mapeados para Case: active, historical (array). O array interactions é usado também para criar registros em Interacao_Ticket_Hugme__c (seção 3).

2. Customer (CustomerBO) → ClientsHugmeIntegration__c

O consumidor do ticket é criado como ClientsHugmeIntegration__c em IntegrationProductHugMeDomain.createCliente(CustomerBO) e associado ao Case via Case.Cliente__c.

Campo Hugme (DTO / API) Campo Salesforce Observação
CustomerBO.nameClientsHugmeIntegration__c.NameNome do cliente.
CustomerBO.email[0]ClientsHugmeIntegration__c.Email__cPrimeiro e-mail do array.
CustomerBO.phone_numbers[0]ClientsHugmeIntegration__c.Telefone__cPrimeiro número do array.
CustomerBO.birthday[0]ClientsHugmeIntegration__c.Data_de_Aniversario__cConvertido via IntegrationProductHugMeUtils.returnDate().
CustomerBO.city[0].nameClientsHugmeIntegration__c.Cidade__cNome da cidade (primeiro elemento do array).
CustomerBO.state[0].fsClientsHugmeIntegration__c.Estado__cSigla do estado (ex.: "SP").

Campos do CustomerBO não mapeados neste fluxo: id, duplicate_id, socialName, cpf, rg, gender, photo, cnpj, company_name, type, tags, pending_tickets_count, city[].id, state[].id, state[].name.

3. Interaction (InteractionBO) → Interacao_Ticket_Hugme__c

Cada interação do ticket vira um registro de Interacao_Ticket_Hugme__c (comentário), criado em IntegrationProductHugMeDomain.createComment(...). Só são inseridas interações que ainda não existem (por ticket_interaction_id) e que possuem message e creation_date.

Campo Hugme (DTO / API) Campo Salesforce Observação
Case.Id (do Case do ticket)Interacao_Ticket_Hugme__c.Ticket__cLookup para o Case.
InteractionBO.ticket_interaction_idInteracao_Ticket_Hugme__c.Ticket_Interaction_ID__cIdentificador único da interação no Hugme (evita duplicatas).
InteractionBO.creation_dateInteracao_Ticket_Hugme__c.Creation_Date__cAjuste de fuso: creation_date.addHours(-3) na criação do comentário; ao persistir, addHours(3).
InteractionBO.messageInteracao_Ticket_Hugme__c.Mensagem__cConteúdo da mensagem.
InteractionBO.ticket_interaction_nameInteracao_Ticket_Hugme__c.Ticket_Interaction_Name__cNome do tipo de interação (ex.: "Manifestação").

Campos do InteractionBO não mapeados para Interacao_Ticket_Hugme__c: ticket_interaction_type_id, customer_id, responsible_id, responsible_name, privacy, modification_date, delivered, readed, visualized, video, picture, details. O objeto possui também Visualized__c no Salesforce (não preenchido por este fluxo).

4. Resposta da API (TicketResponseBO) e outros DTOs

Estruturas usadas na integração que não possuem mapeamento direto para objetos customizados:

Classe DTO Uso
TicketResponseBOEnvelope da API: data (lista de TicketBO) e meta (paginação).
MetaBO / PageBOControle de paginação; não persistido no Salesforce.
HistoricalBO / TypeInfoBOPresentes em TicketBO.historical; não mapeados para objetos SF.
InteractionDetailBODetalhes da interação (interaction.details); não mapeados.
StatusHugmeBO / StatusRaBOFornecem id e name; no Case só é usado o name.
HugmeInteractionResponseResposta ao criar interação/moderação na API; usado para criar comentário no SF (createComment(response, caseId, message)).
PicklistOptionBOOpções de empresa (search companies); exibição em LWC, não persistido como objeto.
